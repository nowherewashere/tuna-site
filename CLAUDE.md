## Build & tooling (Windows, no Node on PATH)
- Portable Node lives at `C:\Users\amdma\node-win\node-v22.22.2-win-x64\`. On the Bash tool, prefix commands with:
  `export PATH="/c/Users/amdma/node-win/node-v22.22.2-win-x64:$PATH"` then use `node` / `npm` / `npx` normally, or call bins directly (e.g. `node ./node_modules/eslint/bin/eslint.js …`). Invoking `npx.cmd` from Git Bash without PATH fails (`"node" is not recognized`).
- Scripts: `dev` = `next dev`, `build` = `next build`, `lint` = `eslint` (flat config `eslint.config.mjs`). **No `typecheck` script** → run `npx tsc --noEmit` (`tsconfig` is already `noEmit:true`).
- **Production is a static export**: `BUILD_STATIC=true next build` → `out/`, served by nginx (Docker `node:22-alpine`). Because of `output:'export'`, `next/image`'s default optimizer is unavailable without `images.unoptimized`.

## Images
- Responsive images use hand-authored `<picture>` + **pre-generated, committed** AVIF/WebP/PNG variants in `public/assets/images/` (naming `name-{W}.{ext}`). `next/image` is intentionally **not** adopted (static export). Variants are generated by a one-off sharp script in the sibling sandbox `tuna-site-audit/scripts/`, not at build time — re-run it if the source art changes. (Wave-1 audit decision, 2026-07-11.)

## Tests / verification
